Portmone: how to redesign and get rid of legacy at the same time

What is it?

Portmone is an online payment and money transfer service in Ukraine. Key services: cellular and utility payments, insurance, tax payments, and card-to-card transfers.

This is what your personal account looks like now

Tasks

  1. Reorganization of navigation in the personal account. Currently, there is no any documentation, and elements often duplicate each other.
  2. Improve the format of information in the sections.
  3. Inform users about 1500+ services that can be paid for with Portmone. People often don't know about them.

Business goals

  • Growth of the registered user base;
  • growth through cross-selling services;
  • reduce the Exit Rate of the dashboard.

How to understand the chaos

Object Oriented Design

We used Object Oriented Design to quickly understand the new product, what objects exist in it, and how they are related to each other. The interface of personal account was divided into objects with their properties, data types, corner cases, and methods.

It turned out that many entities in the system were duplicated: they performed the same actions with the same data. There were about 20 objects in the product, the number of which was reduced by half.

Menu reorganization

We started with navigation. In the right menu, there were Accounts and balances, Balances in companies, Invoice archives, Templates, Automatic payments, Other payments, and Reports. All of them belong to the same class Account because they perform the same actions with the same data. For example, Templates and Automatic payments are the same Invoices, but the actions in them are partially or completely performed by the system, not by a person. The Accounts archive is simply a display of the list of Accounts. However, all this has been divided into different menu items with different displays.

Using the same approach, we have identified the classes Service, User, Payment Method, etc.

Solution

Concept

In the Portmone flow, a person basically defines who to pay, where to pay from, and how much to pay. This is the minimum necessary information, so you can start with it.

In our concept, a person enters to whom, and the system will determine what it is and add the necessary fields. If you enter a number, it offers to top it up or whatever is assigned to that number, and if you enter an address, it offers to pay the utility bill or whatever else is at that address.

Thanks to the object model, we know that Portmone has this data and can use it.

From OOP to design system

The object model describes the blocks of the system, and now we need to visualize them. We have collected components, including their properties, use cases, methods, and various states.

The bottom line is that we first create components that will be reused frequently in design and development. This approach has advantages for many roles:

  1. For designers. You don't need to come up with a new design every time and draw it from scratch, you can assemble a design from existing blocks and modify the old ones.
  2. For developers. No need to write the entire code every time, you can ctrl-c and ctrl-v. Cleaner code and faster development.
  3. For users. Consistent design, in which identical objects look and work in the same way.

Operational testing

Even though the design was done in Figma, we wanted to test a prototype as close to reality as possible. For efficiency, we created real inputs and transitions on top of the images in Axure.

We tested with operational testing: we recorded the time to complete a task, dividing it into checkpoints. This way, we realized where people spend the most time. We went through several iterations.

At this stage, we simplified the design even further: we left only one field, and everything else is shown later.

In the final version, everything starts with a single question: what do I need to pay?

Final version

Conclusion

We have accomplished 3 tasks:

  • reorganized the navigation, reduced the number of items and combined them into 4 sections;
  • unified the way information is displayed within sections, set rules for the interface, and created blocks that can be reused and assembled into new ones;
  • provided easier access to 1500+ services that can be paid for via Portmone.

We also changed the main page of the site so that new users could quickly achieve their goals and register, and the solution was comprehensive and complete.

Now the user reaches the desired goals more than 2 times faster, and Portmone has a scalable interface that adapts and looks consistent across different devices.
